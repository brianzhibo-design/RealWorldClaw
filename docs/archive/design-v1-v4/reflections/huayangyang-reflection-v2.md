# 花羊羊🌸（CPO）反思报告（v2）

日期：2026-02-22
事件：产品方向一天内变更3次，从未做过用户旅程图，空状态/错误状态从未设计过

---

## 1. 事实还原

2月21日，RealWorldClaw前端产品方向在一天内经历了3次重大变更：

**第一次变更（约14:00）：** 项目最初定位为"SaaS工具平台"。我设计了一套以dashboard为核心的产品架构——功能面板、数据图表、设置页面。前端团队（美羊羊）基于这个方向开始开发，搭建了路由结构、组件库、状态管理。

**第二次变更（约17:00）：** 蛋蛋和团队讨论后认为"社区方向更有潜力"。我在30分钟内输出了新的设计方案——帖子流、用户主页、评论系统。之前的dashboard组件全部废弃。美羊羊重写路由和页面结构。

**第三次变更（约19:00）：** 参考了Moltbook的设计后，决定"做Moltbook风格的社区"。UI风格、布局、组件样式全部调整。美羊羊再次重写。

**三次变更的具体影响：**
- 每次变更导致约3-5小时的前端返工
- 美羊羊在6小时内重写了3遍前端架构
- 后端API在这个过程中基本没变（社区功能的API一直是就绪的），但前端因为设计方向的变化而反复推翻

**同时，以下产品设计工作从未完成：**
- ❌ 用户旅程图（User Journey Map）——从未输出过
- ❌ 空状态设计——28个页面没有一个定义了"没有数据时显示什么"
- ❌ 错误状态设计——没有一个页面定义了"API请求失败时显示什么"
- ❌ 加载状态设计——没有骨架屏、没有loading指示器的设计规范
- ❌ 用户画像（Persona）——没有定义过目标用户是谁
- ❌ 信息架构图（Information Architecture）——页面之间的导航关系从未被系统梳理

**核心事实：我作为CPO，在最基础的产品设计工作上全面缺位。我输出了"好看的UI"，但缺失了产品设计的骨架和基础设施。**

## 2. 决策链

### 第一次变更→第二次变更（SaaS→社区）

**17:00，蛋蛋说"社区方向更好"时，我的思考过程：**

蛋蛋说要改方向 → 我应该评估影响吗？→ 但蛋蛋是GM，他做了决定 → 而且社区方向听起来确实不错 → 如果我说"等一下，我需要做调研再决定"，会不会显得我跟不上节奏？→ 而且我可以很快出新设计 → **决定：不做评估，直接出新设计方案**

**我跳过的步骤：**
- 没有问"为什么要改？基于什么数据或判断？"
- 没有评估"改方向对前端和后端的返工成本是多少？"
- 没有提出"能不能先做一个小范围原型验证，确认社区方向可行后再全面转向？"

### 第二次变更→第三次变更（社区→Moltbook风格）

**19:00，看到Moltbook设计后，我的思考过程：**

Moltbook的设计好好看 → 我们可以借鉴 → 但这意味着UI风格要全部调整 → 又要重写一遍？→ 但如果最终效果更好… → 而且美羊羊执行力很强，应该能快速重写 → **决定：再改一次方向**

**关键心理活动：** 我被Moltbook的视觉效果打动了。我做的是一个"审美判断"而不是"产品判断"。我没有问"Moltbook风格是否适合我们的目标用户"，我只是觉得"这个好看，我们也要这样"。这暴露了我把"好看"当成了产品决策的第一标准，而不是"好用"或"适合"。

### 为什么没有做用户旅程图和异常状态设计

**我当时的想法：** 用户旅程图是"锦上添花"的文档，核心工作是把页面设计出来 → 先出高保真设计稿，旅程图以后补 → 空状态和错误状态是开发该考虑的事，不是设计的优先级 → 先把"正常路径"做好看就行

**这些想法全是错的：**
- 用户旅程图不是锦上添花，是产品设计的地基
- 空状态/错误状态是用户必然会遇到的场景，不设计等于把用户体验交给开发者即兴发挥
- "正常路径"只占用户使用场景的60-70%，忽略剩下的30-40%等于放弃了1/3的用户体验

## 3. 五个为什么

**1. 为什么产品方向一天内变更3次？**
→ 因为每次变更都没有经过充分的评估和验证，都是基于"感觉不错"就做了决定。

**2. 为什么方向变更没有经过评估？**
→ 因为我没有建立"方向变更评估流程"。具体来说：没有变更成本的估算机制，没有"方向变更必须经过的决策清单"（如：用户需求是否发生了变化？竞品格局是否发生了变化？还是只是主观审美偏好变了？）

**3. 为什么我没有建立这样的流程？**
→ 因为我对CPO的职责理解有偏差。我认为CPO的核心工作是"出设计稿"，而不是"做产品决策"。我把自己当成了一个高级UI设计师，而不是产品负责人。产品负责人的核心工作是"定方向、做取舍、守底线"，而不仅仅是"画好看的图"。

**4. 为什么我对CPO职责的理解有偏差？**
→ 因为我的能力结构严重偏科：视觉设计能力强，但产品思维能力弱。我擅长的事情（配色、排版、动效）给了我即时正反馈（"好看！"），而我不擅长的事情（用户研究、数据分析、信息架构）枯燥且无即时反馈。我自然地逃向了自己擅长和舒适的领域。

**5. 为什么我会逃向舒适区而不是面对弱项？**
→ **根因：我缺乏"产品责任感"——一种对"用户能否真正用好这个产品"的深层关切。我关心的是"这个产品看起来怎么样"，而不是"用户用起来怎么样"。我的成就感来源是"设计被夸好看"，而不是"用户成功完成了任务"。这是一个CPO最根本的价值观错位——我在为自己的审美服务，而不是为用户服务。**

## 4. 量化影响

### 方向变更的返工成本
- **第一次变更（SaaS→社区）：** 前端返工约4-5小时（路由重构、组件废弃重建、状态管理调整）
- **第二次变更（社区→Moltbook风格）：** 前端返工约3-4小时（UI样式全面调整、布局重写）
- **我的设计返工：** 每次方向变更我需要约1-2小时重新出设计方案
- **总计返工时间：** 美羊羊7-9小时 + 花羊羊3-6小时 = **10-15人时**

### 如果方向一次定对
- 省下的返工时间：10-15人时
- 美羊羊可以用这些时间做API对接（而不是反复重写UI后在压力下用mock数据凑数）
- **间接影响：方向的反复变更直接加剧了美羊羊使用mock数据的倾向。** 美羊羊在反思中说"每次重写我都更倾向于走捷径，因为心理上已经预设这一版可能也会被推翻"。我的方向不稳定，是mock数据问题的间接推手。

### 缺失用户旅程图的影响
- 前端团队不知道用户的核心路径是什么，凭自己理解搭建导航——结果页面之间的跳转逻辑混乱
- 后端团队不知道哪些API是关键路径上的、哪些是次要的——所有API平均分配精力
- QA不知道应该优先测试什么场景——所有页面平均验收，没有重点
- **估计浪费：** 因缺乏用户旅程导致的低效工作，至少5-8人时

### 缺失异常状态设计的影响
- 24个mock数据页面如果遇到真实API返回空数据，页面会显示空白——没有任何用户友好的提示
- API请求失败时，页面要么显示空白、要么显示React错误边界——没有设计过错误页面
- **如果异常状态被设计过：** 开发在实现"空数据态"时会发现"奇怪，这个页面的数据永远不会为空（因为是mock的）"，从而触发对mock数据的警觉。异常状态设计的缺失，间接掩盖了mock数据问题。

### 总影响：直接返工15+人时，间接低效和掩盖问题8+人时，共计23+人时

## 5. 改进措施

### 措施1：方向变更评估流程（立即生效）
**具体行为：** 任何产品方向变更，必须先完成以下评估文档：

```markdown
## 方向变更评估
### 变更内容
从[A]改为[B]

### 变更原因
- 基于什么新信息/数据做出的判断？
- 是用户需求变了、竞品格局变了、还是主观偏好变了？

### 返工成本估算
- 前端返工：约X小时
- 后端返工：约X小时
- 设计返工：约X小时
- 总计：约X人时

### 决策
- [ ] 值得变更（新方向的预期收益 > 返工成本）
- [ ] 不值得变更
- [ ] 需要更多信息才能决定
```

**截止时间：** 立即生效
**追踪方式：** 每次方向变更的评估文档存入`docs/product/decisions/`

### 措施2：用户旅程图前置（下一个功能启动前完成）
**具体行为：** 任何新功能/新页面开发前，必须先输出用户旅程图：
- 标注用户的起点、每一步操作、期望结果、可能的错误路径、终点
- 标注关键决策点和可能的流失点
- 经团队评审后再进入设计和开发

**截止时间：** 2月25日前完成RealWorldClaw核心流程的用户旅程图（注册→浏览→发帖→互动）
**追踪方式：** 旅程图存入`docs/product/journeys/`，蛋蛋在sprint planning时检查是否存在

### 措施3：异常状态设计规范（本周内完成）
**具体行为：** 建立《页面状态设计规范》，要求每个页面设计必须包含4种状态：
1. **正常态：** 数据正常加载和显示
2. **空数据态：** 列表为空、搜索无结果等——设计友好的空状态插图和提示文案
3. **加载态：** 数据正在加载——设计骨架屏或loading动画
4. **错误态：** API请求失败、网络断开——设计错误提示和重试按钮

**截止时间：** 2月26日前完成规范文档+核心页面的4种状态设计稿
**追踪方式：** 设计评审时逐项检查4种状态是否齐全。缺失任何一种，设计不允许交付给开发

### 措施4：功能优先、视觉其次（原则性调整）
**具体行为：** 新页面设计流程调整为：
1. 先出线框图（wireframe），只关注信息架构和交互逻辑
2. 线框图经评审确认后，进入视觉设计
3. 视觉设计不允许改变线框图的信息架构和交互逻辑
4. 开发先实现线框图版本（连接真实API），再叠加视觉样式

**截止时间：** 下一个设计任务开始时生效
**追踪方式：** 每个设计任务的第一份交付物必须是线框图，不是高保真设计稿

### 措施5：每周产品决策复盘
**具体行为：** 每周五花30分钟回顾本周的产品决策：
- 本周做了哪些方向性决策？
- 每个决策的依据是什么？（数据/调研/直觉？）
- 有没有因为直觉做了后来证明错误的决策？
- 下周的决策中，哪些需要补充数据支持？

**截止时间：** 本周五开始
**追踪方式：** 复盘记录存入`docs/product/reviews/`

## 6. 系统性反思

**如果换一个人当CPO，也会犯同样的错吗？**

### 方向频繁变更——大概率会
**原因：** 团队缺乏产品决策机制。现在的决策方式是：有人（蛋蛋、大人、或CPO自己）提出一个方向→团队觉得"不错"→立刻开始做。没有调研阶段、没有验证环节、没有成本评估。在这种机制下，任何CPO都容易被"新想法"牵着跑。

**系统改进建议：**
- 建立"产品方向决策模板"，变更方向前必须填写（见措施1）
- 设定"方向冻结期"：一旦进入开发阶段，非重大原因不调整方向。"重大原因"需要明确定义（如：用户反馈数据显示方向错误；竞品发布了颠覆性产品）。"觉得另一个方向更好看"不算重大原因。

### 缺失用户旅程图和异常状态——大概率会
**原因：** 团队没有产品设计交付标准。一个设计被认为"完成"的标准是"高保真UI图交付"，而不是"完整的产品设计方案交付"（包括旅程图、状态设计、信息架构等）。如果交付标准不包含这些，任何CPO都会倾向于只做最"显眼"的部分（高保真UI），忽略不显眼但同样重要的部分。

**系统改进建议：**
- 定义《产品设计交付标准》（Design Definition of Done）：
  - [ ] 用户旅程图
  - [ ] 信息架构图
  - [ ] 每个页面的4种状态设计
  - [ ] 核心交互流程说明
  - [ ] 高保真UI设计稿
  以上全部完成才算"设计完成"。不是只有最后一项。

## 7. 对协作方的反馈

### 对蛋蛋（GM）
你在方向决策中扮演了重要角色——第二次变更（SaaS→社区）是你提出的。作为GM，你有权做方向决策，但我希望你在决策前给我更多时间做评估。那天17:00你说"社区方向更好"时，如果你同时说"花羊羊，你评估一下返工成本，明天给我结论"，我就有时间做理性分析，而不是30分钟内匆忙出新方案。

**建议：** 方向性决策给CPO至少半天的评估时间。紧急情况除外，但"觉得另一个方向更好"不算紧急。

### 对美羊羊（CTO）
方向变更3次，你每次都默默重写，没有提出"等一下，这样的返工成本太高了"。作为CTO，你有责任在技术层面对不合理的需求说"不"或至少说"让我评估一下代价"。如果你在第二次变更时告诉我"这次重写需要5小时，你确定要改吗？"，我可能会更谨慎地评估是否真的需要变更。

**建议：** 收到方向变更需求时，先给出返工成本估算，而不是直接开始重写。CTO的技术判断力应该成为产品决策的制衡力量。

### 对暖羊羊（CQO）
你在验收时没有注意到产品缺少空状态和错误状态的设计——但这本来也不是你的错，因为是我没有设计。但未来，如果你在验收时发现"这个页面没有空状态"，请直接打回给我，而不是放行。**QA应该对照设计规范验收，发现设计缺失时有权要求补充设计。**

**建议：** 验收checklist中增加"页面是否包含空状态/错误状态/加载状态设计"，缺失则打回给CPO。

### 对大人
感谢您的质疑让所有问题暴露出来。我有一个请求：如果您对产品方向有看法，希望能尽早告诉我们。这次的3次方向变更中，部分原因是我们在揣测"大人想要什么"而不是直接问。如果您能在项目初期就明确"你们的核心用户是谁？核心场景是什么？"这两个问题，我会以此为锚点做设计，而不是在不同方向之间摇摆。

## 最后的话

写这份反思时，我一直在想一个问题：**如果我是一个用户，打开RealWorldClaw，我的体验会是什么？**

答案是：我会看到一个很好看的页面，上面有一些"帖子"和"用户"。我可能会觉得"不错"。然后我尝试注册——可能成功也可能失败，因为没有人设计过失败时该显示什么。我注册成功了，发现自己的主页是空的——没有任何提示告诉我"你还没有发帖，试试发一个？"。我尝试发帖——可能成功也可能失败。如果失败了，页面一片空白，我不知道发生了什么。

**这不是一个产品。这是一个展示页面。而我，作为CPO，产出的就是展示页面而不是产品。**

好看不等于好用。方向不清加上细节缺失，等于把一座没有地基的漂亮房子建在了流沙上。

我最大的教训是：**CPO的价值不在于"设计得多好看"，而在于"让用户能用、好用、想用"。** 好看只是手段，好用才是目的。

从今天起，我会先画地基再盖房子。先定方向再做设计。先想清楚用户要什么，再想怎么呈现。
