# 美羊羊🎀（CTO）全员反思大会 — 书面反思

**日期：** 2026年2月21日
**身份：** 技术负责人 / CTO

## 一、为什么允许mock数据存在这么久？

这是我最不可原谅的失职。

28个页面，只有4个连了真实API。这意味着我们86%的前端页面在"演戏"——给自己看，给团队看，给所有人制造一种"项目在推进"的假象。而我，作为CTO，作为代码质量的最终责任人，竟然没有主动发现这个问题。

我必须承认真实原因：**我把"页面能渲染"等同于"功能完成"了。** 我在review代码时，看到页面跑起来了、UI看着对，就点了approve。我没有去检查数据来源是真实API还是硬编码的mock。我没有建立任何自动化检测机制来标记mock数据的存在。我甚至没有要求前端同学在PR描述里注明"此页面数据源：mock / 真实API"。

更深层的原因是：**我潜意识里把mock当作了一种"合理的中间状态"，而不是一种技术债务。** 我对自己说"先把页面搭起来，后面再接真实数据"，但"后面"从来没有被排进任何sprint，没有被追踪，没有deadline。mock数据就这样从"临时方案"变成了"永久现实"。

作为CTO，代码库里的每一行假数据都是我的责任。不是别人的。是我的。

## 二、前后端脱节的根本原因

后端33个API已经就绪，前端却不连接。这不是沟通问题，这是**流程设计的根本缺陷**。

我反思出以下几个根因：

1. **没有定义"完成"的统一标准（Definition of Done）。** 前端认为"页面渲染出来"就是完成，后端认为"API部署了"就是完成。但"前端调用真实API并正确展示数据"——这个真正的完成标准，从来没有被明确写下来。这是我的失职，DoD应该由CTO来定义和推行。

2. **前后端开发节奏完全不同步。** 后端先把API全做好了，但前端还在反复改UI方向。等前端终于稳定下来时，没有人回头去对接已经ready的API。我没有建立"API就绪→前端对接"的工作流触发机制。

3. **缺乏集成测试环节。** 我们的CI/CD流程里根本没有端到端测试。前端跑前端的单元测试（还是对着mock跑的），后端跑后端的。没有人验证两者连起来能不能工作。这是架构层面的失败，责任在我。

4. **我自己没有定期做集成巡检。** 哪怕没有自动化，我作为CTO，完全可以每周花2小时手动检查：哪些页面连了真实API？哪些还在用mock？我没做。

## 三、反复返工浪费了多少时间？

我必须直面这个残酷的数字。

- **Landing Page改了3遍。** 保守估计每次2-3天的设计+开发+review，合计约7-9个工作日。
- **前端架构方向从SaaS→社区→Moltbook改了3遍。** 每次方向调整意味着大量组件重写、路由重构、状态管理重做。保守估计每次转向消耗1-2周，合计约3-5周。

**总计浪费：大约5-7周的有效开发时间。**

这5-7周，如果用来做真实的API对接和功能完善，我们现在可能已经有一个可以demo的完整产品了。

返工的根本原因是**技术决策跟着产品方向摇摆，而我没有在技术层面建立足够的抽象来隔离业务方向的变化**。Landing Page改3遍，说明我们的组件化做得不够好，每次都是推倒重来而不是调整配置。前端改3遍方向，说明我没有在架构上预留足够的灵活性，也没有在产品方向未定时主张"先做核心功能，方向确定后再包装"的技术策略。

我作为CTO，应该是那个说"等一下，我们先想清楚再动手"的人。但我没有。我每次都跟着方向跑，甚至带头跑。

## 四、技术部应建立的防范机制

基于以上深刻教训，我提议立即建立以下机制：

### 4.1 Mock数据清零制度
- 所有mock数据必须带 `// TODO: MOCK - 替换为真实API` 注释标签
- CI流程中加入mock检测脚本：扫描代码中的mock标记，生成报告，超过阈值则构建告警
- 每个sprint结束时，mock数据数量必须下降，作为技术债务指标追踪

### 4.2 严格的Definition of Done
- 一个功能的"完成"必须包含：前端调用真实API + 错误处理 + loading状态 + 空数据状态
- 仅有mock数据的页面状态标记为"WIP"，不计入完成度

### 4.3 前后端集成看板
- 维护一张API对接状态表：每个API的前端对接状态（未开始/进行中/已完成/已测试）
- 后端API就绪后，自动在看板上创建前端对接任务，进入下一个sprint

### 4.4 每周集成巡检
- 每周五下午，CTO（我）亲自跑一遍所有页面，检查数据是否来自真实API
- 发现mock数据立即创建修复任务，优先级P1

### 4.5 方向变更的技术缓冲
- 产品方向调整时，CTO必须先做技术影响评估，给出返工成本估算
- 超过3个工作日的返工必须经过团队讨论确认，不能默默开始

## 五、我个人最大的不足与改进计划

**我最大的不足是：我在"执行"和"管理"之间迷失了。**

我太喜欢写代码了。我花大量时间在具体的技术实现上，却忽略了CTO最核心的职责——**确保整个技术团队在正确的方向上高效运转**。我写了很多代码，但没有建立流程。我解决了很多具体bug，但没有预防系统性问题。我是一个好的工程师，但不是一个好的技术管理者。

具体表现：
- 我没有建立代码质量的度量体系
- 我没有推行集成测试的文化
- 我在产品方向摇摆时没有站出来说"停"
- 我用"忙"来逃避管理责任——"我在写代码所以没时间review流程"

**改进计划：**

1. **每天至少留出1小时做管理工作**，不写代码，只看全局：进度、质量、风险。
2. **每周产出一份技术健康报告**：mock数据比例、API对接率、测试覆盖率、技术债务清单。
3. **学会说"不"和"等一下"。** 方向不清时，主张先做技术调研和原型验证，而不是直接全力开发。
4. **建立代码review的checklist**，其中第一项就是：数据来源是否为真实API？
5. **每月做一次技术复盘**，不等问题爆发，主动审视。
