## 审查触发条件

### 自动触发
1. **里程碑审查**：每完成一个 P0/P1/P2 阶段
2. **评分变更审查**：项目评分变动 ≥1.0 时
3. **重大技术变更**：架构调整、协议升级时
4. **事件驱动审查**：线上故障、安全事件后
5. **定期审查**：每月第一个周日 18:00

### 手动触发
- 大人指示
- 任何团队成员提出审查请求
- 用户反馈集中某个问题
- 竞品重大变化

## 审查标准化流程

### 第一阶段：数据收集（24h）
```
└── metrics/
    ├── technical-metrics.json     # 代码、测试、性能指标
    ├── business-metrics.json     # 用户、收入、成本数据  
    ├── quality-metrics.json      # bug率、响应时间、可用性
    └── user-feedback.json        # 支持工单、GitHub issues
```

### 第二阶段：分组审查（48h）
每组产出：
- **风险识别**：Top 5 风险点 + 缓解方案
- **改进建议**：3个优先改进项
- **评分**：该领域 0-10分 + 理由

### 第三阶段：综合报告（24h）
慢羊羊🧓整合：
- **项目评分重新校准**（权威）
- **下月优先级**（Top 3 重点）
- **资源分配建议**
- **风险预警**

## 审查输出规范

### 文件命名
```
docs/reviews/YYYY-MM-DD-{trigger}/
├── 00-executive-summary.md      # 执行摘要
├── 01-technical-review.md       # 技术审查
├── 02-business-review.md        # 商业审查  
├── 03-ux-review.md             # 用户体验
├── 04-quality-review.md        # 质量审查
├── 05-recommendations.md       # 综合建议
└── metrics/                    # 量化数据
```

### 评分体系
| 维度 | 权重 | 评分标准 |
|------|------|----------|
| **技术架构** | 25% | 可扩展性、性能、安全、可维护性 |
| **产品体验** | 20% | 易用性、文档质量、学习曲线 |
| **工程质量** | 20% | 测试覆盖、部署稳定性、监控 |
| **商业价值** | 20% | 市场需求、竞争优势、盈利模式 |
| **团队执行** | 15% | 开发效率、协作质量、目标达成 |

## 审查结果跟踪

### Action Items 管理
- **P0**：必须在下次审查前解决
- **P1**：30天内解决  
- **P2**：90天内解决
- **观察**：持续监控但暂不处理

### 改进度量
- **修复率**：上次审查问题的解决比例
- **评分趋势**：连续3次审查的分数变化
- **预测准确性**：风险预警的命中率

## 审查工具化

./scripts/collect-metrics.sh

```

```

### 报告生成
```bash
# 生成审查报告
./scripts/generate-review-report.sh 2026-02-22-milestone
```

## 审查文化

### 原则
1. **客观诚实**：不回避问题，不粉饰太平
2. **建设性**：批评必须伴随改进建议
3. **量化驱动**：用数据说话，减少主观判断
4. **行动导向**：每个问题都要有明确的Action Item
