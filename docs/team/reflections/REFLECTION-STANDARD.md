# 反思规范
> 2026-02-21 · 全员反思不深刻，制定规范后重新写

---

## 一、反思不是检讨书

检讨书 = "我错了，我改" → 空话
反思 = "我在X时刻做了Y决定，因为Z认知，导致W后果，根因是V，改正方法是U" → 具体

## 二、必须包含的要素

### 1. 具体事件（不是抽象总结）
❌ "我允许了mock数据存在"
✅ "2月21日下午，美羊羊提交了Moltbook前端代码，我review时只看了UI截图，没有打开Network面板检查数据来源。这是具体的失职时刻。"

### 2. 决策链还原
❌ "我没有验证"
✅ "当时我的思考过程是：build通过了→页面看起来对→应该没问题。我跳过了'数据从哪来'这个问题，因为我潜意识认为前端工程师会自觉连API。这个假设是错的。"

### 3. 认知根因（挖到底）
用"五个为什么"方法：
- 为什么mock数据上线了？→ 因为没人检查
- 为什么没人检查？→ 因为没有检查流程
- 为什么没有检查流程？→ 因为我认为不需要
- 为什么我认为不需要？→ 因为我把"代码能跑"等同于"功能完成"
- 为什么我有这个等号？→ **因为我从来没有站在用户角度想过"完成"意味着什么**

最后一层才是根因。前面的都是表象。

### 4. 量化影响
❌ "浪费了很多时间"
✅ "Landing Page改3遍，每遍约2天，共浪费6天；前端方向改3遍，每遍约3天，共浪费9天。总计15天有效工作时间被消耗在返工上。如果一次做对，我们比现在快2周。"

### 5. 改进措施（可验证、可追踪）
❌ "以后我会更仔细"
✅ "从明天起，每次前端部署后，我在10分钟内用以下checklist验证：
   - [ ] 打开Chrome DevTools Network面板
   - [ ] 确认首页fetch请求指向真实API（非localhost、非mock）
   - [ ] 点击注册按钮，确认请求发出并收到响应
   - [ ] 截图存入 docs/verification/ 目录
   不做验证不许发布到production。
   追踪方式：每次验证截图带时间戳，weekly review时检查。"

### 6. 系统性反思（不只看自己）
除了个人失职，还要反思：
- 我和上游（谁给我需求）之间的协作哪里断了？
- 我和下游（我的产出给谁）之间的交接哪里断了？
- 公司层面什么机制的缺失导致了这个问题？
- 如果换一个人在我的位置，也会犯同样的错吗？如果会，说明是系统问题不是个人问题

### 7. 对他人的反馈（建设性）
反思不只是自己的事。如果你认为其他部门/同事的某个做法加剧了问题，要说出来：
✅ "蛋蛋在分配任务时只说了'做Moltbook风格的前端'，没有明确交付标准中必须包含'API对接'。如果任务描述中明确写了'禁止mock数据'，我会在第一版就连API。建议蛋蛋在每次任务分配时附上验收清单。"

## 三、格式模板

```markdown
# [姓名] 反思报告
日期：
事件：[具体是什么事]

## 1. 事实还原
[具体时间、具体行为、具体结果，不加修饰]

## 2. 决策链
[当时我在想什么 → 做了什么决定 → 为什么这样决定]

## 3. 五个为什么
1. 为什么发生了X？→ 因为Y
2. 为什么Y？→ 因为Z
3. 为什么Z？→ 因为W
4. 为什么W？→ 因为V
5. 为什么V？→ **根因**

## 4. 量化影响
[浪费了多少时间/多少资源/多少机会]

## 5. 改进措施
[具体到可以验证的行为，有截止时间，有追踪方式]

## 6. 系统性反思
[不只是个人问题，是什么系统缺陷导致的]

## 7. 对协作方的反馈
[对上下游的建设性意见]
```

## 四、反思的评判标准

| 维度 | 不合格 | 合格 | 优秀 |
|------|--------|------|------|
| 具体性 | "我失职了" | 有具体事件 | 有时间、地点、决策细节 |
| 深度 | 1层原因 | 3层为什么 | 5层为什么，挖到认知根因 |
| 量化 | "浪费了时间" | 有大概数字 | 有精确计算 |
| 改进 | "以后注意" | 有具体措施 | 有措施+追踪+验证方式 |
| 系统性 | 只看自己 | 看到流程问题 | 提出制度改进建议 |
| 对他人 | 不提 | 提到了 | 有建设性反馈+具体建议 |
